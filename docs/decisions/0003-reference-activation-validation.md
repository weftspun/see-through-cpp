# 3. Per-component reference-activation validation

Status: accepted (2026-07-17)

## Context and Problem Statement

A ggml reimplementation can be structurally plausible and numerically wrong
(wrong concat order, wrong norm grouping, wrong head split). How do we know
each component is correct before composing the pipeline?

## Considered Options

1. Seeded input/output reference pairs generated by the *upstream* PyTorch
   classes, compared per component (trellis2cpp `Dockerfile.ref` pattern,
   run locally).
2. End-to-end image comparison only.
3. Golden-file unit tests per op.

## Decision Outcome

Option 1. `gen_reference_trans_vae.py` imports `UNet1024` from the
see-through repo itself (not a reimplementation), loads the real
checkpoint, and writes seeded inputs + outputs; the C++ test loads the
GGUF, runs the graph, and reports max/mean abs diff.

- **Tolerance**: max abs diff < 5e-2 gate; observed 6.5e-4 at 256px —
  the f16-weight noise floor. A structural error shows up orders of
  magnitude above this, so the gate is not sensitive to threshold choice.
- **Reference format**: raw binary `{i32 ndim, i64 dims[], f32 data}` per
  array. A minimal npz/zip parser was tried first and abandoned — numpy's
  local-header quirks (data descriptors) segfaulted it, and debugging a
  zip parser is not this project's job. Dumbest format wins.
- **Reduced resolution** (256px vs 1024) is valid because the network is
  resolution-agnostic; it keeps CPU reference generation and validation
  fast enough to run on every change.

Consequences: every future component (VAEs, CrossFrame UNet, Marigold)
gets a `gen_reference_*.py` and a validation target before it is trusted.
End-to-end image comparison (option 2) still happens at pipeline
completion, but it is a smoke test, not the correctness argument.
